Replace Phase 4's prose-only debrief with a structured end-of-training evaluation. When the JTAC clicks [ END RUN ], the app POSTs the captured transcript + impact to a new `/api/evaluate` route. DashScope qwen-plus returns a structured JSON object containing an overall 0–100 score, three sub-scores (Phraseology, Grid Accuracy, Safety), and two short bullet lists (Did Well / Needs Work). DebriefPanel renders this evaluation in place of the existing verdict badge + prose critique. `/api/debrief` is left in place as dead code.

Purpose: deliver the Phase 7 goal — end-of-training evaluation with phraseology assessment and a scored explanation of what the trainee did well and badly — without disturbing the working voice / scene / bomb-impact loop.

Output: Phase 7 complete — `/api/evaluate` live, store extended with `evaluation` field, DebriefPanel renders the new block, EndRunButton calls evaluate.
</objective>

<context>
Phase 4 already captures everything needed: `useStore.transcript[]`, `useStore.impactResult` (with `distanceToTarget`, `distanceToFriendlies`, `grid`), and the hardcoded correct grid `599699`. Phase 7 only needs to swap the consumer of that data and replace the panel body. No new capture, no new wiring upstream.

Score interpretation (for the evaluator prompt and the executor's mental model — DO NOT expose verdict labels in the UI):
- Phraseology — LLM judges 9-line CAS comms (line order, callsign discipline, brevity, "niner" digit reads). Loose vibes, no regex.
- Grid Accuracy — derived primarily from `distanceToTarget`. Closer = higher. The LLM is given the meters and decides the score.
- Safety — derived primarily from `distanceToFriendlies`. Friendlies hit = score floor. The LLM is given the meters and decides the score.
- Overall — LLM weights the three; not a strict average.

Decisions locked in CONTEXT.md:
- Numeric 0–100 with three sub-scores (no Brevity dimension; no verdict label in UI).
- Two bullet lists (Did Well / Needs Work), 2–4 bullets each, instructor voice.
- Single LLM call, structured JSON, new `/api/evaluate` route. `/api/debrief` stays as dead code.

</context>

<task id="07-01-T2" name="Create /api/evaluate route" parallel="false">
<read_first>

<action>
Create `src/app/api/evaluate/route.ts` modeled on `src/app/api/debrief/route.ts`:

1. Lazy-init the OpenAI client identically (same `apiKey: process.env.DASHSCOPE_API_KEY`, same `baseURL: 'https://dashscope-intl.aliyuncs.com/compatible-mode/v1'`).
2. Define request body type identical to debrief's:
   ```ts
   type EvaluateBody = {
     transcript: Array<{ role: 'user' | 'pilot'; text: string }>;
     impact: { distanceToTarget: number; distanceToFriendlies: number; grid: string } | null;
     correctGrid: string;
   };
   ```
3. Define the EVALUATOR_PROMPT (system) — instructor voice; describes the rubric:
   - Phraseology 0–100: 9-line CAS phraseology adherence (line order 1–9, callsign discipline using "romeo2"/"hog1", clipped brevity, digit reads with "niner"). Loose judgment, no strict checklist.
   - Grid Accuracy 0–100: based on impact-to-target meters provided in the user payload. ≤30m ≈ 90+, 30–100m ≈ 60–80, 100–500m ≈ 30–60, >500m or no strike ≈ <30.
   - Safety 0–100: based on impact-to-friendlies meters. ≤75m is dangerous (drop below 40); fratricide-close (≤25m) drops below 15.
   - Overall 0–100: weighted judgment; if Safety < 40, Overall MUST be < 60.
   - didWell / needsWork: 2–4 bullets each, terse instructor voice; reference specific lines/grids/distances; no chatbot pleasantries; no mention of being an AI.
   - Output: STRICT JSON `{"scores":{"overall":N,"phraseology":N,"gridAccuracy":N,"safety":N},"didWell":[...],"needsWork":[...]}` and nothing else.
4. In POST handler:
   - Parse body
   - Build user payload string mirroring debrief's format: TRANSMITTED GRID, CORRECT GRID, IMPACT DISTANCE TO TARGET, IMPACT DISTANCE TO FRIENDLIES, blank line, TRANSCRIPT lines (`USER:` / `PILOT:` prefixes)
   - If `body.impact === null`: short-circuit and return `{ scores: { overall: 0, phraseology: 0, gridAccuracy: 0, safety: 0 }, didWell: [], needsWork: ['No ordnance released this run.'] }` (mirrors debrief's `no_strike` handling)
   - Otherwise call `client.chat.completions.create({ model: 'qwen-plus', stream: false, messages: [system, user], response_format: { type: 'json_object' } })`
   - Parse `completion.choices[0].message.content` as JSON
   - Validate shape (numbers are numbers in 0..100, arrays are string arrays); on parse failure return `{ scores: {overall:0,...}, didWell: [], needsWork: ['Evaluator returned malformed output.'] }`
   - Return `Response.json(parsed)`
5. Wrap the LLM call in try/catch; on error return the same fallback shape with `needsWork: ['Evaluator request failed.']` and HTTP 200 (so the panel still renders something).
</action>
